# amori-infra — принципы работы

Проектные принципы личной AI-инфраструктуры. Когда возникает выбор «как сделать» —
сверяйся с этим документом.

## 1. Два контура (personal / customer)
Личные данные CEO и персональные данные клиентов РФ — это **разные требования**.
- **Personal** (агенты, Obsidian, dev, память) — гибкость, размещение на Mac Mini.
- **Customer** (leads, support, переписка) — ПДн граждан РФ, цель: РФ-резидентность (152-ФЗ).
Сырые ПДн клиентов не пересекают границу — личные агенты получают только агрегаты.

## 2. Tier-routing моделей по «ставке» задачи
- Tier-1 (Claude Pro / ChatGPT Plus, ручной) — high-stakes: стратегия, сложный код, креатив.
- Tier-2 (платный API, под бюджет-гардом) — mid-stakes автоматизация.
- Tier-3 (Groq free / Ollama local) — bulk, рутина, приватное.
Правило: дорогая модель только когда дешёвая объективно не тянет (`router.py`).

## 3. Fail-safe defaults (отказ безопасен)
- Бюджет-гард: исчерпан лимит платного API → **даунгрейд на free**, не блок (`cost_guard`).
- Импорт cost_guard в router защищён try/except: проблема с ops_db **не** ломает роутинг.
- Бэкап best-effort: падение компонента понижает статус, но не валит весь прогон.
- Любой агент должен переживать недоступность внешнего сервиса (Ollama → fallback Groq).

## 4. Наблюдаемость обязательна, тишина — это сигнал
- Каждый LLM-вызов → `ops_db.llm_usage`. Ручной Tier-1 → `tier1_sessions` (раньше был «слепым»).
- Каждая инфра-задача пишет `infra_runs` + `infra_heartbeats` (dead-man's-switch).
- Алертим в Telegram **только при проблемах** (`notify.py`), но heartbeat пишем всегда —
  отсутствие heartbeat само по себе детектируется монитором.

## 5. Бэкап без off-site и без restore-теста не считается бэкапом
- Риск #1 — отказ внутреннего SSD. Поэтому off-site на **отдельный физический диск**
  (внешний USB) обязателен; копия верифицируется контрольными суммами.
- Ежемесячный `restore_test` поднимает одноразовый postgres и реально восстанавливает —
  только пройденный restore подтверждает, что бэкап живой.
- Нет внешнего диска и нет restic/B2 → бэкап шлёт WARN (молча «без off-site» недопустимо).

## 6. Изоляция операционных данных
Observability/учёт живут в отдельной БД `ops_db`, не смешиваясь со схемой Langfuse в `agents`.
Новые app-данные — туда же или в customer-контур, не в общую `agents`.

## 7. Single-responsibility агенты
Каждый агент делает одно и хорошо. Координация — через общие хранилища и (план) n8n,
а не через монолит. Новый агент = новый файл + launchd plist + запись в INFRA.md.

## 8. Секреты — из окружения, не из кода
Токены в `.env` (права 600), не хардкодом. Цель — machine secret-backend (Infisical/OpenBao)
с ротацией и audit. До миграции — минимум: ничего нового не хардкодить, `.env` под бэкапом.

## 9. Ресурсы конечны — следи за диском и логами
Mac Mini 24/7 переполняется незаметно: логи растут, Docker копит образы/кэш.
- Логи > 10MB → ротация (gzip+truncate) в `backup.sh`.
- Монитор алертит при < 10GB (warn) / < 5GB (crit) на Data-томе и при раздувании Docker.
- `df /` на macOS обманчив (запечатанный сис-том) — мерить по `$HOME` (Data-том).

## 10. Изменения — обратимо и проверяемо
Правки в прод-инфру: аддитивно, с проверкой, не ломая работающее. Рискованное
(переименование БД, перенос данных) — отдельным ревьюемым шагом, не «по дороге».
